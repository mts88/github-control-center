# Red-Team Review — Marketplace Rollout Plan (VS Marketplace + Open VSX)

Plan under review: automate publishing to VS Marketplace and Open VSX through the existing semantic-release pipeline (`publishCmd` on the exec plugin), plus one-time account/secret setup.

## Summary

The plan is directionally sound but the single proposed `publishCmd` is its weakest point: it puts secrets on the command line, relies on shell features (`&&`, `$VAR` expansion) that `@semantic-release/exec` does not guarantee, and orders registry publishing *before* the GitHub Release — so the most likely failure (a bad/expired token on the very first run) strands a half-published release with no GitHub Release artifact to recover from. All fixable with token env-var conventions, split exec plugin instances, and a pre-flight token check.

## By severity

- **Critical**: None.

- **High**:
  1. **Tokens on the command line** — `-p $VSCE_PAT` / `-p $OVSX_PAT` exposes secrets to failure paths: when the command fails, execa (used by `@semantic-release/exec`) includes the full command line in the error, which lands in the public Actions log. GitHub masks exact secret matches, but relying on masking for a public repo is fragile. Both CLIs natively read `VSCE_PAT` and `OVSX_PAT` from the environment — drop `-p` entirely and pass the secrets only as step env vars.
  2. **`publishCmd` assumes a shell** — `$VSCE_PAT` expansion and `&&` chaining only work if the exec plugin runs the command through a shell; its execution model does not guarantee this. If not, the first release fails (or worse, passes garbage as an argument). Fix: no `$VAR` at all (see point 1) and one command per plugin instance — `@semantic-release/exec` can be listed twice, once per registry — instead of `&&`.
  3. **Publish ordering creates the worst partial-failure mode** — with the exec plugin listed before `@semantic-release/github`, registry publishing runs *first*. If vsce succeeds and ovsx (or the GitHub Release) fails, the tag and release commit are already pushed but there is no GitHub Release with the vsix, and semantic-release will not re-release on workflow re-run (the release commit is `[skip ci]` and the tag exists). Invert the order: add a *second* exec plugin instance **after** `@semantic-release/github` carrying only the `publishCmd`s — then the durable artifact (GitHub Release + vsix) always exists before any registry is touched, and marketplace recovery is a documented manual `--packagePath` publish of that exact asset.

- **Medium**:
  1. **Unpinned `yarn dlx` at publish time with secrets in env** — `yarn dlx @vscode/vsce` fetches the latest version on every release; a breaking release or a compromised package runs with both marketplace tokens in its environment. Pin exact versions (`@vscode/vsce@x.y.z`, `ovsx@x.y.z`) or add both as devDependencies so Yarn's lockfile owns them. The existing `prepareCmd` has the same flaw; fix it in the same PR.
  2. **No pre-flight token validation** — the first end-to-end test of both tokens is a real release on `main`. Add a Phase 0 verification: `npx @vscode/vsce verify-pat mts88` and `npx ovsx verify-pat mts88` locally after creating the tokens, so a 401 never burns a release.
  3. **Listing ships without screenshots** — the README is the marketplace page and is text-only. For a "control center" extension the tree views + PR panel screenshot is the selling point. Not a blocker, but the rollout plan's stated goal is presentation.

- **Low**:
  1. `--no-dependencies` is a packaging flag; it is meaningless on `vsce publish --packagePath` — drop it from the publish command to avoid implying it does something.
  2. `galleryBanner` may be ignored by the current marketplace UI (post-redesign); harmless to set, but don't count it as visible polish.
  3. No rollback note: `vsce unpublish` / Open VSX admin removal exist but are destructive and cached downstream — worth one line in CLAUDE.md so nobody improvises.
  4. Marketplace propagation is not instant on first publish (verification/scan can take minutes); Phase 2 verification should say "wait up to ~15 min" to avoid a false alarm.
